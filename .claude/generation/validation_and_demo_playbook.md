# Validation And Demo Playbook

## Purpose
Maximize submission strength by ensuring the generated application is both internally validated and easy for a judge to understand during inspection.

## Internal Validation Goals
- the app boots
- the run lifecycle is visible
- agents display distinct roles
- the final recommendations can be traced backward
- retries and repairs are visible when they occur

## Judge-Facing Demo Goals
- it is obvious one prompt triggered the whole system
- it is obvious multiple agents collaborated
- it is obvious the architecture is modular
- it is obvious code was generated by workflow rather than prewritten

## Required Demo Story
1. Show the initial prompt entering the system.
2. Show the orchestrator planning and dispatching agents.
3. Show domain agents producing artifacts.
4. Show frontend observability surfaces rendering trace data.
5. Show final synthesis with evidence and rationale.

## Guardrails
- do not rely on a judge reading internal files to understand the app
- the generated frontend must carry the demonstration burden visually
